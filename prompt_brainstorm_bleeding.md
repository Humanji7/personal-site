# Brainstorm: Bleeding Effect 2.0

## Контекст

Сфера теперь **катится** за курсором (mesh rotation). Старая механика bleeding не работает:
- Частицы падают в **локальном** пространстве, но mesh вращается → они "зависают" визуально
- После падения частицы телепортируются обратно, а не возвращаются плавно

---

## Текущая реализация (проблемная)

```
1. Frantic movement → Trigger bleeding
2. Select random particles → Add tremor (jitter)
3. Mark as "falling" (type=2) → Apply gravity in LOCAL space
4. If y < -2.5 → Teleport back to original position
```

**Проблемы:**
- Gravity в локальном пространстве ≠ визуально "вниз" когда mesh повёрнут
- Нет плавного возврата к сфере
- Падающие частицы не участвуют в rolling

---

## Идеи для обсуждения

### Вариант A: World-Space Gravity
- Конвертить позиции частиц в world-space перед применением gravity
- Сложнее, но физически корректно

### Вариант B: "Испарение" вместо падения
- Частицы не падают, а **растворяются** (alpha fade out)
- Респавн: fade in на случайной позиции сферы
- Проще, элегантнее?

### Вариант C: "Орбитальный отрыв"
- Частицы не падают вниз, а **отлетают** от центра (radially)
- Потом медленно возвращаются по спирали
- Работает независимо от rotation

### Вариант D: Shader-based bleeding
- Весь эффект в шейдере через uniform `uBleedIntensity`
- Частицы "проваливаются" к центру или наружу
- Полностью GPU-based, без JS-манипуляций positions

### Вариант E: Оставить только tremor (без падения)
- При frantic movement — только дрожь/jitter
- Никакого falling, просто визуальный стресс
- Самый простой вариант

## Вопросы для размышления

1. Какой **эмоциональный смысл** у bleeding? Что оно должно передавать?
2. Должны ли частицы **физически** покидать сферу, или достаточно визуального эффекта?
3. Как bleeding должен взаимодействовать с rolling? Или они независимы?
